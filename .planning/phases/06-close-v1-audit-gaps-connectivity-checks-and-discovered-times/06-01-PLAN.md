<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI process -> external replay source | Untrusted network response crosses into source connectivity check. |
| CLI process -> S3-compatible storage | Storage credentials and bucket metadata are used by a read-only probe. |
| CLI process -> PostgreSQL staging database | Database URL and staging schema visibility are tested through SQL. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Information Disclosure | `src/check/source-connectivity.ts` | mitigate | Discard source body and return only controlled `status`, `failureCategory`, and `message` fields. |
| T-06-02 | Tampering | `src/check/s3-connectivity.ts` | mitigate | Use only `HeadBucketCommand`; unit tests grep out `PutObjectCommand` and `DeleteObjectCommand`. |
| T-06-03 | Tampering | `src/check/postgres-connectivity.ts` | mitigate | Use only constant read SQL `select 1` and `select 1 from ingest_staging_records limit 1`; no SQL interpolation or writes. |
| T-06-04 | Denial of Service | `src/check/postgres-connectivity.ts` | mitigate | Always close owned `Pool` with `pool.end()` in `finally` so CLI scripts do not hang. |
| T-06-05 | Elevation of Privilege | `src/check/*` | mitigate | No parser imports, no RabbitMQ, no canonical server-2 business-table writes; static acceptance criteria enforce boundary. |
</threat_model>
