# Plan 02-02: Diagnostics, Failures, and Pacing

<objective>
As an operator, I want dry-run discovery to distinguish item diagnostics from source-level failures and respect cautious request pacing, so that source instability does not silently corrupt candidate evidence or hammer the replay source.
</objective>

<threat_model>
## Threat Model

- T-02-03: Source-level failure is reported as a successful empty run. Mitigation: source unavailable and rate-limited failures set `ok: false` and CLI exit code 2.
- T-02-04: Discovery polls too aggressively and triggers source protection. Mitigation: sequential detail fetches and injectable delay with a default 2000 ms minimum between source requests.
- T-02-05: Diagnostics leak secrets or become untestable prose. Mitigation: use stable diagnostic codes and source URLs only; no credentials are included.
</threat_model>

<action>
Implement structured diagnostics and failure handling:
- Missing filename on a detail page produces diagnostic `{ code: "missing_filename", severity: "warning" }` and skips that candidate.
- Malformed rows without a replay link produce `{ code: "malformed_row", severity: "warning" }` and continue.
- Duplicate filenames produce `{ code: "duplicate_filename", severity: "warning" }` and preserve both candidate evidence rows unless exact candidate objects are identical; do not auto-merge as product truth.
- Changed metadata for the same filename/source ID within one run produces `{ code: "changed_metadata", severity: "warning" }`.
- Source fetch failures that look unavailable produce `{ code: "source_unavailable", severity: "error" }`, `ok: false`, and no thrown crash for expected network/source errors.
- Source fetch failures that look like Cloudflare/rate-limit/HTTP 429 produce `{ code: "rate_limited", severity: "error" }`, `ok: false`.

Update CLI behavior so `discover --dry-run` sets `process.exitCode = 2` when `report.ok === false`, while item-level warnings with `ok: true` exit 0.
</action>
