## High 🟠

3. `src/run/run-once.ts:592–597` [observability] — `writeFinalCheckpoint` catches errors without logging the `error` object, while the F1 fix in the same file explicitly added `error` to the `writePageCheckpoint` warn (line 562). Introduced by this change: the F1 patch repaired the per-page checkpoint log but left the final-checkpoint catch inconsistent. A persistent backend failure (e.g. CAS headers rejected on every write) is diagnosable from `writePageCheckpoint` logs but invisible at the final write — exactly the scenario `conditionalWrites: false` was added to handle.

   Fix: add `error` to the final catch log, mirroring the per-page pattern:
   ```ts
   } catch (error) {
     input.log?.warn(
       { error, slug: context.slug },
       "final checkpoint write failed; continuing run",
     );
   }
   ```
   `[conv: §D — observability §AA; std: correctness §AA]`

## Medium 🟡

1. `src/errors/app-error.ts:6` [quality] — Docstring says _"derives `name` from the concrete subclass"_, but the mechanism changed in this PR: the base now hardcodes `this.name = "AppError"` and each subclass must explicitly set `this.name` in its own constructor. The old `new.target.name` did it automatically; the new pattern requires an explicit per-subclass override, which the existing four subclasses all do correctly (this diff adds them). The stale docstring creates a convention trap for future subclass authors — a new `extends AppError` without an explicit `this.name` will silently surface as `"AppError"` in logs and serializers.

   Fix: update the docstring to reflect the explicit-override contract:
   ```ts
   * Each concrete subclass MUST set `this.name` explicitly in its constructor
   * (e.g. `this.name = "SpecificError"`) — the base hardcodes `"AppError"` and
   * does not auto-derive from `new.target.name`.
   ```
   `[std: §A — naming; std: correctness → Code-quality / Comments]`

2. `src/commands/discover.ts:98–100` and `src/commands/run-once.ts:69–71` [error-system] — Both defensive guards throw a raw `new Error(...)` for a programmer-bug condition (staging repository unexpectedly `undefined`). Per `[conv: §D]`, unexpected non-`AppError` exceptions from action bodies propagate through commander's `parseAsync` as unhandled promise rejections and exit non-zero — so the process does crash correctly. However, the error bypasses the typed error system and the conventions comment (`// logged at fatal with the error object`) — a future CLI-level error handler catching `instanceof AppError` would not categorize these correctly.

   These paths are `/* v8 ignore next */` and unreachable in practice (both callers always pass `shouldStage: true`). The fix is low-urgency but aligns with the error taxonomy:
   ```ts
   // Instead of: throw new Error("Expected staging repository for run-once");
   // Create an AppError subclass or (minimally) keep the raw Error but document
   // it as an intentional programmer-bug guard, not an operational error.
   ```
   `[std: §B — typed error system; conv: §D — CLI error boundary]`

## Non-Findings Checked

- **`unwrapCause` + AbortError/TimeoutError (F2):** Traced both fetch-error paths. When `fetch()` aborts via `AbortSignal.timeout()`, the thrown `DOMException` (name `TimeoutError` or `AbortError`) has no `.cause`, so `unwrapCause` reads `current.name` on the exception itself — correctly routed to `transient`. When the error is wrapped in a cause chain by `source-client.ts` (line 145, `{ cause: input.originalError }`), `unwrapCause` descends to `.cause` and reads `cause.name` — also correct. Both call sites in `replay-byte-client.ts` pass the raw DOMException directly. No body-leak: `name` is a short static string, not response-body content. Fix is correct.

- **`booleanFromEnvironment` default for `conditionalWrites`:** When `S3_CHECKPOINT_CONDITIONAL_WRITES` is absent, `optional()` returns `undefined`, the transform returns `true` — so CAS headers stay on by default. Correct safe default; unconditional fallback requires explicit `false`.

- **`AppError` subclass `this.name` completeness:** All four concrete subclasses in scope (`SourceFetchError`, `ReplayByteFetchError`, `CheckpointConflictError`, `ConfigValidationError`) set `this.name` explicitly in this diff. No existing subclass was missed.

- **Mechanical func-style refactor (`function foo(){}` → `const foo = ()=>`):** No hoisting/TDZ issues found. All converted functions are module-level `const` expressions referencing only already-declared identifiers; no circular reference patterns across the conversion. `this` binding irrelevant (no method-style calls).

- **`src/run/types.ts` barrel:** Re-exports exactly the same set of types that were previously defined inline. No type-widening or omissions. The import fence direction is fixed (adapters in `evidence/` now import `types/run-summary.ts` downward, not `run/` upward).

- **`ConfigValidationError` constructor:** Takes `issues: string[]` (not `ZodIssue[]`). Both call sites in `config.ts` already map ZodIssues to strings via `.map(issue => ...)` before constructing. No change in the wire contract for callers consuming `error.issues`.

- **`conditionalHeader` logic in `s3-checkpoint-store.ts`:** When `conditionalWrites: false`, returns `{}` (empty spread) — PutObjectCommand gets no conditional headers. When `true` and `etag` is `undefined` (first write), returns `{ IfNoneMatch: "*" }`. When `true` and `etag` is set, returns `{ IfMatch: etag }`. CAS semantics correct and unchanged from the original.
