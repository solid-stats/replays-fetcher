# Quick Task 260623-x57: Skip per-row detail fetch in watch discovery for already-staged replays - Context

**Gathered:** 2026-06-23
**Status:** Ready for planning

<domain>
## Task Boundary

The watch loop's per-cycle wall-clock is dominated by the per-row detail-page
fetch in discovery: `discoverPageCandidates` (`src/discovery/discover-dedup.ts`)
iterates page-1 list rows and, for every row, calls `discoverRowCandidate`
(`src/discovery/discover-candidate.ts`) which fetches the detail HTML solely to
extract `filename`. Each detail read is serialized under the
`REPLAY_SOURCE_REQUEST_SPACING_MS` floor (default 250ms), so a ~60-80 row page
costs ~15-20s/cycle.

The DEDUP-01 pre-fetch gate added in v3.1 (phase 24-03, `src/run/ingest-page.ts`)
only skips the **byte download + staging** for already-staged `externalId`s. It
runs in `ingestPage`, AFTER discovery has already paid the detail fetch for every
row. The dedup key (`source.externalId`) is already present on the list row
BEFORE the detail fetch — so for an already-staged replay the detail fetch is
wasted work.

**Goal:** Gate the detail fetch on the same existence check, so the watch cycle
collapses to roughly (new replays × spacing) + one list read instead of (all
page-1 rows × spacing). Steady-state cycles drop from ~15-20s to sub-second.

</domain>

<decisions>
## Implementation Decisions

### Where the gate lives — predicate threaded into discovery
- Thread an OPTIONAL `existsBySourceIdentity` predicate (plus `sourceSystem`)
  into the discovery input. Inside the page-row loop in `discoverPageCandidates`,
  BEFORE calling `discoverRowCandidate`, if the row's `externalId` is trustworthy
  AND already staged, skip the row — the detail HTML is never fetched.
- run-once does NOT pass the predicate, so its discovery path is byte-for-byte
  unchanged (no DB coupling for run-once).
- The predicate is an injected async function (NOT a `pg` import), mirroring how
  DEDUP-01 already threads `existsBySourceIdentity` — discovery stays pure and
  unit-testable with a fake predicate. Chosen over a two-phase
  list-parse/detail-resolve split in the watch loop because it is a smaller diff,
  one-pass, and reuses the existing injection pattern.

### Cannot-miss guard preserved (Claude's discretion, locked by precedent)
- Reuse the SAME trustworthiness rule as the existing gate (`isTrustworthyId`,
  `src/run/ingest-page.ts:109`): only a non-empty, non-whitespace `externalId`
  may skip pre-fetch. Absent/empty/whitespace ids MUST still fetch the detail —
  their post-fetch `source_replay_id` is a `derived:` form that needs the
  downloaded checksum and cannot be matched pre-fetch. This is the data-loss
  guard; it must not regress.

### Keep the post-fetch DEDUP-01 gate (Claude's discretion)
- The existing `ingestPage` pre-fetch dedup gate STAYS as defense-in-depth: it
  still catches the derived-id case (untrustworthy externalId that survived
  detail fetch but is already staged) and any race between cycles. The new
  discovery-level skip is an optimization layered IN FRONT, not a replacement.

### Skip legibility (Claude's discretion)
- A discovery-level skip must remain observable in the run summary / diagnostics
  the same way `skippedBySourceId` is surfaced today (§AA skipped-vs-processed
  legibility). The planner decides the exact counter/diagnostic shape; the
  invariant is that a skipped-pre-detail row is tallied as skipped, never as
  discovered/failed/malformed.

</decisions>
