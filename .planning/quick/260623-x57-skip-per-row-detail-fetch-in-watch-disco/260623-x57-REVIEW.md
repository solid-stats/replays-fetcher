## Info 🔵
1. `src/discovery/discover-dedup.ts:197` [async] — `await shouldSkipPreDetail(...)` issues one `existsBySourceIdentity` DB SELECT per row inside the sequential row loop — structurally an N+1 read. The inline comment justifies it as "the deliberate pre-detail gate against a rate-limited source," but the predicate hits PostgreSQL (the staging table), not the rate-limited HTTP source, so that specific framing is imprecise. It is **not a real defect**: the watch path is page-1-only (`WATCH_PAGE`/`maxPages = 1`, bounded row count), the gate genuinely must run before the per-row fetch decision, and it mirrors the already-accepted per-candidate `existsBySourceIdentity` check in `ingest-page.ts:205`. No fix required; if the comment is ever revised, retag the rationale as "bounded page-1 existence gate that must precede the fetch decision," not "rate-limited source."
2. `src/run/run-once-page.ts:47` [legibility] — `appendDiscoveryReport` hardcodes `skippedPreDetail: 0` rather than copying `pageReport.counts.skippedPreDetail`. Correct today (run-once can never opt into the predicate, so the page report's value is always 0) and the comment says so, but it is a defensive constant that would silently mask a future value if run-once ever gained the predicate. Reading `pageReport.counts.skippedPreDetail` would be equally safe and self-maintaining. Purely cosmetic — not blocking.
