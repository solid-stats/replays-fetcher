<objective>
Add an always-on `watch` daemon runtime mode to `replays-fetcher` for sub-minute discovery of newly-published replays, running ALONGSIDE the existing scheduled `run-once`. The watcher polls source page 1 every ~15s (configurable), ingesting new replays within seconds instead of waiting for a scheduled full crawl. This is a deliberate, user-approved override of the prior "scheduled run-once only, no always-on crawler" runtime shape.

The watcher is page-1-only and CHECKPOINT-INDEPENDENT: it never reads or advances the source checkpoint (that belongs to run-once's full crawl). It reuses the existing per-page ingest machinery (discover page 1 ã store raw S3 ã write staging) via a shared, checkpoint-free helper, so no ingest logic is duplicated. A single cycle failure is logged and the loop continues; only a config error (exit 2) or unrecoverable fatal stops it. On SIGTERM/SIGINT the loop stops cleanly, flushes pino, sets `process.exitCode = 0`. Each successful cycle writes a heartbeat file for a k8s exec liveness probe (NO HTTP server ã conventions forbid a web framework).

Purpose: ingest new replays within seconds of publication; a separate nightly full run-once (infra, out of scope) catches anything off page 1.
Output: `watch` command + `watch-loop` orchestration + shared `ingest-page` helper + bounded Zod config + conventions/architecture record update.
</objective>
