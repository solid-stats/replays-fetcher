<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator CLI flags → run behavior | `--emit-evidence`/`--evidence-file` toggle a durable write surface |
| in-memory RunSummary → stdout (compact) | the projection decides what reaches the operator/server-2-visible stdout contract |
| process exit → buffered log stream | final NDJSON lines must drain before exit |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-05 | Information Disclosure | compact stdout projection | mitigate | stdout is `toCompactSummary(...)` only — the heavy arrays cannot reach stdout under any flag (asserted by the cli test with --emit-evidence set). The compact document keeps only identifier/scalar fields. |
| T-11-08 | Repudiation | dropped final log lines on exit | mitigate | The cli action owns a single awaited `flushLogger(rootLogger)` Promise, called AFTER the stdout write and BEFORE `process.exitCode`; `process.exit()` is never used (the destination is synchronous and drains). Asserted by the flush-ordering test (D-16). |
| T-11-03 | Tampering / boundary breach | opt-in S3 evidence write surface | accept (scoped) | The evidence store shares the existing bucket/credentials; the write is opt-in (default off) and log-and-continue in run-once (Plan 03). The cli adds no other write surface; no staging-schema/raw-key/server-2 change. The contract doc records the cross-app-visible stdout change (Compatibility Rule). |
| T-11-10 | Tampering | operator `--evidence-file` path → local disk | accept | First local-disk write in the project; operator owns the path and cleanup (D-13). The write is gated on the flag, log-and-continue, and never a default path. |
| T-11-SC | Tampering | npm/pip/cargo installs | mitigate | No package installed in this phase. `--evidence-file` uses `node:fs/promises` (stdlib). No legitimacy gate applies. |
</threat_model>
