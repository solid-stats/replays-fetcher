<objective>
Close the real ARCH-05 gap: the watch daemon currently exits WITHOUT draining its pg.Pool or
destroying its S3Client. `createStoreRawResources` builds both (`shared.ts:219` for s3Client; the
pool inside `createStagingRepository` at `:207-209`) but `StoreRawResources` (`:86-93`) exposes
neither ŌĆö so `watch.ts` physically has no handle to close them. pg.Pool keeps the event loop alive,
which on a long-lived k8s daemon = a connection leak on every pod-termination (SIGTERM).

Fix: expose a once-guarded `dispose()` from `StoreRawResources` (a closure capturing s3Client + pool
that runs `s3.destroy()` + `await pool.end()` exactly once), and call it from `watch.ts`'s existing
`finally`, AFTER `runWatchLoop` resolves (the drain point ŌĆö in-flight ingest is already done there).
Reuse the existing `createShutdownSeam()` signal handler ŌĆö do NOT register a second process listener.

Purpose: ARCH-05 ŌĆö clean k8s pod-termination; no hanging pool/sockets; teardown owned by the
composition root, never an adapter.
Output: `dispose()` on `StoreRawResources`, wired into `watch.ts`; extended watch unit tests with
injected fake clients; a new multi-cycle watch teardown integration test.
</objective>
