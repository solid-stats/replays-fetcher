## High 🟠

3. `src/commands/shared.ts:218–224` [resource-lifecycle / §AA] — `createDispose` swallows errors
   silently on both teardown legs. If `s3Client.destroy()` throws, the exception is caught by the
   `finally` clause that runs `pool?.end()`, but **neither leg logs the failure**: a teardown error
   disappears with no trace, making it undiagnosable in production. The `pool?.end()` rejection (if
   any) also surfaces as an unhandled rejection from the `async` closure with no log.

   `[std: correctness §AA — swallowed errors; §AB — resource lifecycle]`

   **Fix:** add a `try/catch` around each teardown call (or a single outer catch) that logs the
   error before the closure resolves:

   ```ts
   const createDispose = (
     s3Client: S3Client,
     pool: Pool | undefined,
     log: Logger,
   ): (() => Promise<void>) => {
     let disposed = false;
     return async (): Promise<void> => {
       if (disposed) return;
       disposed = true;
       try {
         s3Client.destroy();
       } catch (err) {
         log.warn({ err }, 'S3Client.destroy() threw during teardown');
       }
       try {
         await pool?.end();
       } catch (err) {
         log.warn({ err }, 'pg.Pool.end() threw during teardown');
       }
     };
   };
   ```

   Note: `createDispose` currently takes no logger. Either thread the `rootLogger` from
   `createStoreRawResources` (already available at the call site) or accept a `log` parameter.
   The watch command's `finally` block is the correct call site — the logger is already flushed
   afterward, so the warn will not be lost.

## Medium 🟡

1. `src/discovery/discover-candidate.ts:63–75` [§AA — swallowed errors] — `parseSourceFixture`
   has a silent `catch { return undefined; }` with no log. If the JSON parse fails, the caller
   (`discoverPageCandidates` in `discover-dedup.ts`) falls through to HTML-based discovery with
   no record of why the fixture was discarded. In production a malformed fixture is
   indistinguishable from a missing one.

   `[std: correctness §AA — swallowed errors; severity table: swallowed error → 🟠]`

   Classification rationale: downgraded from 🟠 to 🟡 because this is the fixture/dry-run path,
   not the live ingest hot-path, and the caller has a valid fallback — the run is not silent-broken,
   only the diagnostic is lost. Raise to 🟠 if this path ever runs in scheduled `run-once` mode.

   **Fix:**
   ```ts
   export const parseSourceFixture = (
     text: string,
     log?: Logger,
   ): SourceFixture | undefined => {
     try {
       const parsed = JSON.parse(text) as Partial<SourceFixture>;
       if (Array.isArray(parsed.candidates)) {
         return { candidates: parsed.candidates };
       }
     } catch (err) {
       log?.warn({ err }, 'fixture JSON parse failed; falling back to HTML discovery');
       return undefined;
     }
     return undefined;
   };
   ```

2. `src/run/run-once-checkpoint.ts:167–180` (approximately; `writePageCheckpoint` catch block)
   [§AA — traceback] — the `catch` block logs `{ error, ... }` using the key `error` rather than
   `{ err, ... }`. Pino serializes the error object with its built-in `err` serializer only when
   the key is literally `err`; using `error` logs the error as a plain object and **discards the
   stack trace**.

   `[std: correctness §AA — traceback preserved on logged exceptions]`

   Evidence gate: `log?.warn({ error, ... }, ...)` at the catch site — the error object is
   referenced but not under the `err` key.

   **Fix:** rename the binding and the log field:
   ```ts
   } catch (err) {
     input.log?.warn(
       { err, page: page.lastCompletedPage, slug: page.slug },
       'checkpoint write failed; continuing run',
     );
     return page.etag;
   }
   ```
   Apply the same rename to `writeFinalCheckpoint`'s catch block for parity.

  conventions note "raw artifacts meant to be durable are covered by an explicit retention
  decision"). Leg 3 of the §AB test (no cleanup and no retention policy) is present in the
  planning record. Not a finding.

  cleanup). No double-free of the same resource. Ruled out.

  `disposed` flag is set synchronously before any `await`, so a concurrent second call sees

  the W-01 fix from the prior review — verified present. Ruled out.
